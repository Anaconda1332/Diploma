# 1.4 Выводы

Проведённый анализ показал, что на рынке представлены как коммерческие, так и open-source решения для обнаружения мошенничества в мобильных приложениях. Коммерческие SDK и облачные сервисы обеспечивают быструю интеграцию и поддержку, однако часто ограничены в гибкости и требуют значительных финансовых вложений. Open-source библиотеки предоставляют больше возможностей для кастомизации, но требуют высокой квалификации и самостоятельной поддержки.

Для эффективной работы системы обнаружения мошенничества в мобильных приложениях необходима реализация следующих принципов:

- Автоматизация анализа поведения пользователя и событий в приложении;
- Гибкая настройка сценариев реагирования;
- Многоуровневая система уведомлений для пользователей и разработчиков;
- Простота интеграции и масштабируемость решения;
- Высокий уровень защищённости и минимальное влияние на производительность.

Анализ подходов к машинному обучению для скоринга риска (п. 1.4 аналитической части) позволил сформулировать дополнительные выводы:

- Задачу оценки риска мошенничества целесообразно формулировать как **бинарную классификацию** с выходом в виде вероятности p ∈ [0, 1], что даёт возможность задавать градации риска (низкий / средний / высокий) и гибко настраивать сценарии реагирования.
- Признаковое пространство может формироваться из данных, уже собираемых SDK (root, контакты, Wi‑Fi, биометрия, версия ОС, звонок, время), с сохранением фиксированного набора и порядка признаков при обучении и на устройстве.
- **On-device ML** (выполнение модели непосредственно на устройстве) с использованием **TensorFlow Lite** снижает задержку, экономит трафик и повышает конфиденциальность, так как сырые признаки не передаются на сервер при расчёте скора.
- Таким образом, выбранная стратегия — комбинация собственного SDK для сбора сигналов и встроенной ML-модели для оценки риска — согласуется с лучшими практиками рынка, обеспечивает гибкость и расширяемость решения и позволяет адаптировать систему под конкретные требования и сценарии использования.

# 1.5 Цель и задачи УИР

Целью данной учебно-исследовательской работы является разработка и внедрение программного модуля (SDK) для обнаружения мошеннических действий в мобильных приложениях на платформе Android, обеспечивающего своевременное выявление подозрительных событий и информирование пользователей и разработчиков.

Для достижения поставленной цели необходимо решить следующие задачи:

1. Провести анализ существующих решений и определить оптимальные подходы к построению системы обнаружения мошенничества.
2. Сформулировать требования к функционалу и архитектуре разрабатываемого SDK.
3. Разработать архитектуру SDK, обеспечивающую интеграцию с Android-приложениями.
4. Реализовать основные алгоритмы обнаружения подозрительных действий и событий.
5. Внедрить механизмы уведомления пользователей и разработчиков о выявленных инцидентах.
6. Провести тестирование SDK на предмет корректности работы, производительности и удобства интеграции.
7. Подготовить документацию и рекомендации по использованию SDK для сторонних разработчиков.

Дополнительные задачи, связанные с ML-модулем оценки риска:

8. Провести анализ подходов к машинному обучению для скоринга риска мошенничества (бинарная классификация, признаковое пространство, on-device ML, TensorFlow Lite).
9. Спроектировать и реализовать модель оценки степени риска (MLP, бинарная кросс-энтропия, метрика AUC), обучить модель на подготовленных данных и конвертировать её в формат TensorFlow Lite для выполнения на устройстве.
10. Спроектировать и реализовать модуль оценки риска (Risk Scorer) в составе SDK: загрузка TFLite-модели и параметров нормализации из ресурсов приложения, формирование вектора признаков из данных SDK, расчёт risk score и градаций риска (низкий / средний / высокий), интеграция в поток обнаружения и в логику приложения (уведомления, сценарии реагирования).

# Выводы по главе 3 (Инженерная часть)

Архитектура системы обнаружения мошенничества организована по принципу модульной многоуровневой модели, что способствует логическому разделению компонентов и повышает масштабируемость. Вся система разделена на два основных уровня: уровень SDK и уровень приложения. Уровень SDK представлен модульной библиотекой, реализованной с использованием паттерна Singleton и принципов Clean Architecture, и отвечает за сбор данных о состоянии устройства и потенциальных рисках мошенничества. Уровень приложения, построенный на базе Android с использованием MVVM-архитектуры и ViewBinding, обрабатывает пользовательские взаимодействия, реализует управление разрешениями и обеспечивает отображение данных через современный Material Design интерфейс.

SDK для обнаружения мошенничества включает шесть функциональных модулей, каждый из которых отвечает за определённый аспект безопасности. Модуль сбора информации об устройстве использует системные API Android для получения технических характеристик, включая производителя, модель, версию операционной системы и другие параметры. Модуль обнаружения root-доступа выполняет комплексную проверку устройства на наличие признаков получения root-привилегий через анализ системных тегов сборки и проверку стандартных файлов. Модуль биометрической безопасности интегрируется с системными API для проверки доступности биометрических методов аутентификации. Модуль мониторинга телефонных вызовов отслеживает состояние активных вызовов в реальном времени. Модуль анализа контактов предоставляет информацию о количестве контактов в адресной книге пользователя. Централизованная система управления разрешениями обеспечивает корректную работу всех модулей, автоматически определяя необходимые права доступа и предоставляя разработчикам информацию о недостающих разрешениях.

В рамках проектирования (п. 3.3) дополнительно спроектирован модуль оценки риска (Risk Scorer), образующий седьмой функциональный блок SDK. Модуль обеспечивает загрузку TFLite-модели и параметров нормализации из ресурсов приложения (assets), формирование вектора признаков из данных перечисленных модулей SDK в фиксированном порядке, расчёт числового risk score (0.0–1.0) и определение градации риска (низкий / средний / высокий). Интеграция Risk Scorer в поток обнаружения позволяет приложению на основе скора выбирать сценарий реагирования (пропуск операции, запрос дополнительной аутентификации или блокировка) и информировать пользователя через интерфейс и уведомления.

Android-приложение с интеграцией SDK реализует комплексную систему обработки разрешений и отображения данных. Пользовательский интерфейс организован в виде модульной системы карточек, каждая из которых отвечает за отображение определённого типа информации, получаемой от SDK. Основная активность приложения корректно управляет жизненным циклом Android-компонентов, обеспечивая сохранение состояния при повороте экрана и временном переключении между приложениями. Система включает в себя функции для принудительного обновления данных, что полезно для тестирования различных сценариев. Интерфейс приложения адаптируется к статусу разрешений, показывая соответствующую информацию о доступности различных функций и предоставляя пользователю понятные инструкции по их предоставлению. Карточка «ML Risk Score» и кнопка «Рассчитать риск» обеспечивают вызов модуля оценки риска и отображение числового скора и градации риска, что завершает интеграцию проектируемой архитектуры в единый пользовательский сценарий.

# Выводы по главе 4 (Технологическая и практическая часть)

Разработанный SDK для обнаружения мошенничества успешно реализован в виде Android-библиотеки с использованием паттерна Singleton, что обеспечило эффективное управление ресурсами и предотвращение создания множественных экземпляров. Архитектура SDK логически разделена на модули для различных аспектов безопасности: анализ устройства, проверка root-доступа, биометрическая аутентификация, мониторинг звонков и анализ контактов. Использование современных Android API, включая BiometricManager и TelephonyManager, позволило создать надёжную систему обнаружения потенциальных угроз безопасности.

Демонстрационное Android-приложение построено с использованием Material Design и ViewBinding, что обеспечило современный и отзывчивый пользовательский интерфейс. Карточная структура интерфейса с использованием CardView улучшила читаемость информации и создала интуитивно понятную навигацию. Интеграция SDK в приложение реализована через Gradle-зависимости, что упростило процесс разработки и обеспечило тесную связь между компонентами. Система управления разрешениями работает автоматически, запрашивая необходимые права доступа при запуске и обрабатывая ответы пользователя.

В рамках технологической части реализован модуль оценки риска на основе машинного обучения (п. 4.7, 4.8). Подготовка учебных данных выполняется скриптом генерации синтетических признаков; обучение модели MLP и оценка AUC выполняются скриптом на Python (Keras, scikit-learn). Обученная модель конвертируется в формат TensorFlow Lite и вместе с параметрами нормализации размещается в ресурсах SDK (assets). Модуль FraudRiskScorer загружает модель и scaler из assets, формирует вектор признаков из данных SDK, выполняет инференс на устройстве и возвращает risk score и градацию риска (низкий / средний / высокий). В демо-приложении карточка «ML Risk Score (TensorFlow Lite)» и кнопка «Рассчитать риск» обеспечивают вызов расчёта риска и отображение результата, что подтверждает работоспособность on-device скоринга в составе единого сценария использования.

Проведённое ручное тестирование на эмуляторе Android Studio подтвердило стабильность и корректность работы всех компонентов системы. Тестирование охватило инициализацию приложения, управление разрешениями, функциональность SDK и пользовательский интерфейс. Все основные функции работают корректно: сбор информации об устройстве, определение root-статуса, анализ биометрии, мониторинг звонков и подсчёт контактов. Проверена работа модуля оценки риска: при наличии модели и параметров в assets расчёт risk score и отображение градации риска выполняются без ошибок; при отсутствии файлов выводится понятное сообщение пользователю. Интерфейс отзывчив и адаптируется к различным размерам экранов, а система обработки ошибок обеспечивает стабильную работу приложения даже при отказе в предоставлении разрешений.
